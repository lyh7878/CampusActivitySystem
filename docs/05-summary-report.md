# 校园活动报名与签到管理系统 - 项目实验总结报告

## 1. 项目组织情况
本项目由 **linxingyu** 一人独立完成，模拟了4人团队的协作流程。

我通过在不同的 Git 分支上工作，依次扮演了以下角色并完成了相应任务：
-   **项目经理**：负责仓库初始化、目录结构建立、README编写和最终成果合并。
-   **计划负责人**：在 `feature/wbs` 分支上完成了三级WBS分解和进度计划编制。
-   **配置管理员**：在 `feature/config` 分支上制定了配置管理方案。
-   **协作成员**：通过 `feature/conflict-A` 和 `feature/conflict-B` 两个分支，模拟两人修改同一文件，成功制造并解决了合并冲突。

所有角色的工作成果均通过分支合并的方式集成到 `dev` 分支，并最终合并至 `main` 分支提交。

## 2. WBS分解思路
-   **分解原则**：严格遵循"结果导向"，自上而下，围绕项目的最终交付物进行分解。
-   **层级设计**：L1为成果域（如项目管理文档、系统需求），L2为子成果（如项目计划），L3为最小可管理单元（如WBS文档本身）。
-   **验证**：采用100%规则，确保所有L3条目的总和完整覆盖了项目范围。

## 3. 进度安排依据
-   **活动来源**：所有活动均从WBS的L3级成果单元直接提取，确保工作不遗漏。
-   **依赖关系**：遵循项目工作流的自然顺序，如先需求分析后设计，再编码再测试。
-   **工期估算**：基于单人工作负荷进行估算，文档类工作通常1天，设计类2天。通过并行安排（如文档撰写和编码）来压缩总工期。

## 4. 配置管理过程
-   **配置项识别**：共识别8大类配置项，包括所有项目文档和示例代码。
-   **分支策略**：严格执行 `main`、`dev`、`feature/*` 的三层分支模型。所有特性开发在 `feature` 分支进行，集成验证在 `dev` 分支，最终交付通过合并到 `main` 完成。
-   **提交规范**：所有提交信息遵循 `type(scope): description` 格式，保证了日志清晰可读。
-   **标签使用**：在最终提交完成后，通过 `v1.0` 标签标记了实验结束点。

## 5. 冲突制造与解决过程
-   **冲突制造方法**：本实验为单人成组，我通过以下步骤模拟了多人协作的冲突场景：
    1.  在 `dev` 分支建立基线文件 `docs/04-conflict.md`。
    2.  从 `dev` 分别创建 `feature/conflict-A` 和 `feature/conflict-B` 两个分支。
    3.  在两个分支上，分别对文件的同一位置（第一段落）做出不同修改并提交。
    4.  切回 `dev` 分支，先合并A，再尝试合并B，人为制造了合并冲突。
-   **冲突解决过程**：Git自动标记了冲突位置，我手动编辑了 `04-conflict.md` 文件，去除了冲突标记 `<<<<<<<`、`=======` 和 `>>>>>>>`，并结合双方修改意图，生成了最终的整合版本。
-   **最终结果**：冲突被成功解决，合并后的文件通过提交 `merge: 解决冲突...` 正式纳入 `dev` 分支，整个过程被完整记录在Git历史中。

## 6. 实验收获与问题
-   **收获**：
    1.  深刻理解了WBS"结果导向"的意义，能区分"动作"和"成果"的区别。
    2.  掌握了Git分支协作的核心工作流，特别是解决冲突的实战方法。
    3.  认识到配置管理计划对于保证项目一致性和可追溯性的重要作用。
-   **遇到的问题**：
    1.  单人模拟多人协作时，需要清晰规划分支创建次序，避免自己逻辑混乱。实际操练中，执行 `git merge feature/wbs` 时因分支未正确切换而出现 `Already up to date` 的情况，后通过 `git checkout -b dev` 创建并切换分支解决。
    2.  编写符合规范且简洁的提交信息，需要刻意练习。
