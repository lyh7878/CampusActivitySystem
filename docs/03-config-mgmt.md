# 校园活动报名与签到管理系统 - 配置管理方案

## 1. 配置项识别
下表列出了项目中需要纳入版本控制的主要配置项（CI）。

| 配置项名称 | 存储路径 | 类型 | 说明 |
|---|---|---|---|
| 项目概况文件 | `README.md` | 文档 | 项目介绍与协作约定 |
| WBS文档 | `docs/01-wbs.md` | 文档 | 范围计划交付物 |
| 进度计划文档 | `docs/02-schedule.md` | 文档 | 进度计划交付物 |
| 配置管理文档 | `docs/03-config-mgmt.md` | 文档 | 本文件 |

| 实验总结报告 | `docs/05-summary-report.md` | 文档 | 最终实验报告 |
| 示例源代码 | `src/` | 代码 | 项目示例代码 |
| 测试示例文件 | `test/` | 代码 | 项目测试示例 |

## 2. 分支策略
采用简化的Git Flow模型，分支命名与用途如下：

- **`main`**：受保护的主分支。只接受来自 `dev` 的合并，存放经过审核的、随时可交付的最终成果。
- **`dev`**：开发集成分支。各个特性分支的工作首先合并到此处，进行集成验证。
- **`feature/*`**：临时功能分支。基于 `dev` 创建，完成后合并回 `dev`，并删除。例如：
    - `feature/wbs`：用于编写WBS和进度计划
    - `feature/config`：用于编写配置管理方案
    - `feature/conflict-A` 和 `feature/conflict-B`：用于制造冲突

## 3. 提交、合并与版本标记规则
- **提交信息规范**：采用"type(scope): 简短描述"格式。例如：
    - `feat(wbs): 完成三级WBS分解`
    - `docs(readme): 更新协作约定`
    - `merge: 合并feature/wbs到dev`
- **合并规则**：向 `dev` 或 `main` 分支的合并，必须在本地完成，并解决可能出现的所有冲突。不允许强制推送。
- **版本标记**：在最终提交至 `main` 分支后，创建一个轻量标签 `v1.0` 作为实验结束的标志。
